# 问题 Framing 模板

```markdown
## 问题 framing

### 主题核心问题
- 当前项目真正想回答的问题：
- 这个问题的输入、输出和评估对象：
- 这个问题不是：

### 当前常见假设
- 输入假设：
- 任务假设：
- 模型能力假设：
- 评测假设：
- 数据分布假设：
- 计算资源假设：

### 主题边界
- 明确包含：
- 明确不包含：
- 暂时不碰的相邻方向：
- 必须覆盖的代表论文或 baseline：

### 用户偏好和资源约束
- 想做偏方法 / 评测 / 系统 / 应用 / 分析：
- 风险偏好：保守 / 中等 / 激进：
- 可用算力和数据：
- 期望周期：
- 目标 venue 或组内目标：

### 为什么这个问题值得研究
- 学术价值：
- 实用价值：
- 为什么不是简单工程优化：
- 如果解决失败，是否仍有负结果价值：

### 如果要收缩成可做的小问题
- 一个更小的问题版本：
- 最小可验证设置：
- 最小可实现方法：
- 最小 baseline 集合：
- 最小指标集合：
- 一周内能完成的 sanity check：

### 需要补证据的问题
- 哪个 claim 还不确定：
- 应该用 paper-finder 搜什么：
- 搜到什么结果会支持这个方向：
- 搜到什么结果会否掉这个方向：

### Hypothesis
- 核心假设：
- 为什么已有论文暗示这个假设可能成立：
- 哪个实验最能证伪它：
- 如果被证伪，下一步怎么改：
```

## 使用提醒

- 先把问题说清楚，再谈 gap。
- 很多“创新点”只是 framing 不清导致的幻觉。
- 好的 framing 应该能自然导出 experiment plan，而不是只导出漂亮标题。
